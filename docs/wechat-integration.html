01TL;DR · 一分钟看懂

把微信变成 pi 的命令行前端。用户在微信里发一条消息 → 微信消息落到腾讯 iLink 服务器 → pi-web 后台 3 秒一次的轮询把消息拉回来 → 喂给本地的 pi coding agent → 把 agent 的回复发回微信。 整个过程不经过任何第三方服务器中转（除了腾讯自己）。

⌬

协议

腾讯 ilinkai.weixin.qq.com 的 iLink Bot CGI （/ilink/bot/*），Bearer Token + 随机 X-WECHAT-UIN。

∞

模型

不绑定模型 —— 复用 lib/rpc-manager.ts 启动 AgentSession，模型、工具、思考级别全部从 settings.json / 入参继承。

◍

职责

仅做 消息搬运工 + 微信账号托管；不实现鉴权配额、不做媒体处理 （图片/语音只记录联系人，没有回复通路）。

关键事实：pi-web 不依赖 任何微信第三方库。 third/openclaw-weixin/ 是一份完整的腾讯 @tencent-weixin/openclaw-weixin v2.4.3 源码拷贝（用于参考协议），但 运行时根本没引用。 lib/wechat/ 是按需重新实现的精简版。

02为什么要做这个

pi-web 本来是个 浏览器里的 pi 聊天 UI。要做到"在微信里跑 agent"，需要解决三件事：

  1. 消息通道 —— 个人微信没有官方 Bot API，但腾讯为 OpenClaw 平台开放了 iLink Bot CGI 协议。 pi-web 直接对接这套 CGI，把"扫码登录 → 收发消息"做成自家后端。
  2. 会话连续性 —— 同一个发信人在微信里连发 3 条消息，应该在 同一个 coding session 里 （同一个 .jsonl 文件）。否则每次发消息都开新会话，agent 完全没有上下文。
  3. 超时与重试 —— LLM 跑长任务可能 30 秒 ~ 几分钟，微信的同步会话不允许挂那么久。 必须 fire-and-forget：先回个"正在处理"，跑完再单独发一条。

三个核心问题对应的设计选择

问题设计选择落在哪个文件
消息从哪儿进 3 秒短轮询 getUpdates，配 sync_buf 游标 lib/wechat/monitor.ts
会话怎么绑定 account.json 持久化 currentSessionId，/new 主动重置 lib/wechat/state.ts + inbound.ts
同步回微信 订阅 agent_end 事件 → 5 分钟超时兜底 → 取最终 assistant 文本 lib/wechat/inbound.ts

左边是用户手机，右边是腾讯 iLink 服务器，中间是 pi-web（Next.js）。 三方之间的 唯一通道 是 HTTPS POST —— 微信在浏览器侧只负责一个 QR 扫码。

04登录流程 · QR 状态机

登录用 客户端轮询 而不是 WebSocket：用户点"开始登录" → 后端拉一张 QR → 前端 2 秒一次问后端"扫上了没" → 后端去问 iLink → 状态推进。

时序

用户 (UI) 点 "开始登录" → POST /api/weixin/login
pi-web 后端 get_bot_qrcode → iLink (固定 base URL)
iLink 返回 { qrcode, qrcode_img_content: "weixin://..." }
pi-web 后端 QR.toDataURL(weixin://) → base64 PNG
pi-web 后端 state.createSession({qrcode, baseUrl, ...}) → 内存中保留 6min
用户 (UI) 拿到 PNG → 展示
用户 (UI) 每 2s 轮询 → GET /api/weixin/login?sessionKey=...
pi-web 后端 get_qrcode_status → iLink
iLink 返回 { status: "wait" | "scaned" | "need_verifycode" | "confirmed" | ... }
pi-web 后端 advancePhase() → LoginSession.phase 状态机
用户 (UI) 前端根据 phase 切换 UI

关键代码 · advancePhase 状态推进

// app/api/weixin/login/route.ts:107
function advancePhase(session, status) {
  switch (status.status) {
    case "wait":            return set({ phase: "waiting" });
    case "scaned":          return set({ phase: "scanned" });
    case "need_verifycode": return set({ phase: "verifying", message: "需要输入配对码" });
    case "scaned_but_redirect": {
      const newBase = status.redirect_host
        ? `https://${status.redirect_host}` : session.baseUrl;
      return set({ phase: "redirected", baseUrl: newBase });
    }
    case "confirmed": {
      const account = {
        accountId: status.ilink_bot_id, token: status.bot_token,
        baseUrl:   status.baseurl || session.baseUrl,
        userId:    status.ilink_user_id,
        savedAt:   new Date().toISOString(),
        status:    "ok" as const,
      };
      state.saveAccount(account);
      api.notifyStart({ baseUrl, token }).catch(() => {});
      return set({ phase: "confirmed", account });
    }
    case "expired": return set({ phase: "expired", message: "二维码已过期" });
    default:       return session;  // 未知状态保留旧值
  }
}

05消息轮询 Monitor

登录完成后，后台默默跑一个 3 秒一次的轮询器。设计极其克制：refcount-free 单例， 没有线程，没有锁，3 秒一 setTimeout。

tick 循环伪代码

// lib/wechat/monitor.ts
async function tick() {
  timer = null;
  if (inFlight) { scheduleNext(1000); return; }   // 上一轮还没回来？跳过
  inFlight = true;
  try {
    const account = state.loadAccount();
    if (!account) return;   // logout 后自停

    const resp = await api.getUpdates({
      baseUrl: account.baseUrl, token: account.token,
      getUpdatesBuf: syncBufs.get(account.accountId) ?? "",
      timeoutMs: 3000,
    });

    if (resp.get_updates_buf) syncBufs.set(account.accountId, resp.get_updates_buf);

    for (const msg of (resp.msgs ?? [])) {
      const preview = extractPreview(msg);   // 从 item_list 里抠第一条 text
      state.recordContact(account.accountId, msg.from_user_id, preview, msg.context_token);
      if (preview) {
        void handleInbound({ fromUserId, text: preview, contextToken });  // fire-and-forget
      }
    }
  } catch (err) { log.warn("tick failed", err); }
  finally {
    inFlight = false;
    scheduleNext(1000);   // 1s 间隔 + 3s 长轮询 = 平均 ~4s/轮
  }
}

🎯 为什么不长轮询？

腾讯 iLink 支持 longpolling_timeout_ms 最多 35s。 选 3s 短轮询是 故意的：连接生命周期短，方便热重载和进程重启后秒级恢复。 真正"零消息时也在轮询"而已，CPU/带宽都可忽略。

🧠 sync_buf 游标

服务端返回的 get_updates_buf 是不可见的不透明字符串， 每次带回去就行。存 Map<accountId, buf>，只在内存， 进程重启会丢 —— 但 iLink 端似乎能容忍短暂丢游标，新一轮拉空后会再发。

06入站消息处理（核心）

handleInbound() 是整个微信渠道的灵魂。315 行代码，把一条微信消息变成 agent 回复。

图 3 · handleInbound 的 6 步流水线

为什么只看 agent_end？

pi 编码 agent 在一次 turn 里会喷出 几十个 事件：turn_start、 message_update（每个 token）、tool_execution_start、 tool_execution_end... 不同 LLM provider（Anthropic / OpenAI / Google） 喷出的 message_update 形状还不一样。 agent_end 是唯一一个跨 provider 稳定的"回合结束"信号，且必带完整 messages[] 快照。 所以代码里只监听它，从 messages 末尾倒着找最后一个 assistant message，把所有 text 块 join 起来 —— 完事。

07Session / Workspace 语义

三件需要记住的事

L2 · 切 Workspace 必开新会话

用户从下拉框选了一个新 cwd，后端 setCurrentWorkspace(id) 时 同时清掉 currentSessionId。 下一条微信消息落到新目录里，没有旧上下文。

B2 · 越早发"正在输入"越好

进 handleInbound 的第一时间就 void fireTyping()， 不 await。微信 UX 上能看到"对方正在输入"，避免用户以为没收到。

N1 · /new 不冷启动

/new 只清 currentSessionId + 回"✅ 已重置"。 不 立即开新会话 —— 下一条消息才冷启动，并成为新会话的 第一条。 这样 /new 本身永远不会被 agent 看到。

09数据存到哪儿

整套微信渠道在磁盘上只有 3 个文件，且都放在 ~/.pi-web/wechat/。 session 文件（.jsonl）走的是 pi 自己的目录，不归 微信模块管。

路径内容权限生命周期
~/.pi-web/wechat/account.json accountId / token / baseUrl / userId / currentWorkspaceId / currentSessionId / status chmod 600 登录时写入，logout 时清空
~/.pi-web/wechat/sessions.log JSONL 审计：inbound / cold_start / send / agent_event / agent_end / agent_error / reply / reply_failed chmod 600 append-only，永不删除
~/.pi/agent/sessions/<encoded-cwd>/<ts>_<uuid>.jsonl 真正的 agent 会话流（不在微信模块下） 默认权限 由 pi-coding-agent 自身管理
安全要点：account.json 含 bot_token， 等于"以你的微信身份发消息"的能力。一定 chmod 600。 整个项目 没有 微信 refresh_token 机制 —— token 失效就 re-scan QR。

10关键设计决策（避坑指南）

从代码注释和 AGENTS.md 抠出来的 6 个"看似随意实则刻意"的选择 —— 改了会出 bug。

❶ 架构 in-process，不走 HTTP

inbound.ts 直接 import rpc-manager 而不是 fetch('/api/agent/...')。注释原话："the listening port varies (dev=30141, prod=14514) and a hardcoded fallback silently broke the channel when the two diverged."

❷ 时序 notifyStart 必须在 boot 重发

重启 pi-web 后，iLink 会把"长时间没心跳"的 bot 当作离线、停止入队消息。 instrumentation.ts 一启动就调一次 notifyStart， 配合 startup.ts 的 30s rescan 兜底。

❸ UX 越早 sendtyping 越好

进入 handleInbound 的第一步就 void fireTyping()， 不 await。失败被吞掉（debug log）—— 这是个 best-effort 提示，agent 跑 30s 期间用户能看到"对方正在输入"。

❹ 协议 agent_end 是唯一真理

每个 LLM provider 的 streaming 事件形状都不一样（message_update），但 agent_end 始终带完整 messages[]。代码里只 listen agent_end，从不尝试拼 streaming。

❺ 语义 /new 不冷启动

用户发 /new → 只清 currentSessionId + 回"已重置"。 下一条 消息才冷启动、并作为新会话的 第一条。 保证 /new 这个字符串永远不会进 agent 的 prompt 历史。

❻ 错误 401/expired → 自动 mark

safeReply 捕获 sendmessage 失败时，如果错误文本里含 401|expired|invalid|token，就 state.markAccountExpired()。 下次 /api/weixin/status 时面板会显示"账号已过期"的红色 banner。

iLink HTTP 协议形态（一份就够）
